# **Cursor Interaction Rules for the Esoteric Digital Library Project**

**Goal:** Define clear and consistent cursor behaviors across the platform’s UI to reinforce intuitive interactions and accessibility. By changing the cursor style (e.g., **`pointer`**, **`text`**, **`grab`**) in context, the interface communicates which elements are interactive, aligning with usability best practices and accessibility standards. Below are cursor interaction guidelines for key interface components.

## **Knowledge Graph View**

- **Node Hover & Click:** When the cursor hovers over a graph node, highlight that node and its directly connected edges to emphasize related content. The cursor should switch to a **`pointer`** (hand icon) on node hover, indicating the node is clickable. Clicking a node opens its corresponding note or detail view, so maintaining a **`pointer`** on hover prepares the user for this interaction. (Right-clicking a node can open a context menu of node-specific actions; within that menu, each action item should also show a **`pointer`** on hover to indicate clickability.)
- **Node Drag (Repositioning):** If nodes can be manually repositioned in the graph, the cursor should indicate this capability. On hovering a movable node (or a drag handle, if provided on the node), use a **`grab`** cursor to suggest the node can be dragged. Once the user clicks and holds the node, change the cursor to **`grabbing`** to confirm the element is being dragged. This direct-manipulation cue makes it clear the user is physically moving the node, reinforcing an intuitive, tactile experience.
- **Canvas Panning:** The graph background (empty canvas) should support click-and-drag panning for navigation. When the cursor is over an empty area of the graph, display a **`grab`** cursor, signaling that the user can drag the canvas. On click-and-drag of the background, switch to a **`grabbing`** cursor until release. This approach (similar to dragging a map) provides a consistent visual affordance for moving around the graph. Standard scroll-wheel or trackpad gestures can be used for zooming, with no cursor change needed.
- **Interactive Edge or Icon Cues:** If the graph includes interactive icons (e.g. a filter or lens icon in the corner of the graph view) or if edges can be clicked for more info, ensure those elements also use **`pointer`** on hover. All interactive UI elements within the graph view should visibly respond to hover – for example, by a slight glow or the project’s accent color – in addition to the cursor change, to make the interactive regions obvious. Use the platform’s designated accent (the “old gold” color) for hover highlights on graph icons or buttons to keep the style consistent and accessible.

## **Block-Based Editor (Personal Grimoire)**

- **Text Editing Areas:** Within any text block (paragraph, heading, list item, etc.), use the **`text`** cursor (I-beam) when hovering or focusing the content area. This cursor should appear as soon as the user moves into editable text, indicating that text can be selected or edited. The I-beam cursor provides a clear signal for editing, ensuring the user knows they can type at that location (a key expectation in any text editor).
- **Block Hover & Drag Handle:** Each content block reveals a drag handle (e.g. the ⋮⋮ icon) when the block is hovered. On hovering this drag handle, the cursor should become a **`pointer`** to show it’s an interactive control. The drag handle has two functions: (1) **Context Menu:** A normal click on it opens a menu of block options (e.g. “Delete”, “Duplicate”, “Turn Into”). (2) **Drag-and-Drop:** Clicking and holding the handle initiates a drag action to reposition the block. During the drag operation, switch the cursor to **`grabbing`** to visually confirm the block is being moved. This hand-grab cursor, combined with real-time block movement, gives the user tactile feedback, mirroring Notion’s fluid block rearrangement experience. After dropping the block in a new location, return the cursor to default or whatever is appropriate for the element now under the cursor.
- **Slash “/” Command Menu:** Typing "/" in a new line brings up an inline block-type menu for power users. While this menu is keyboard-driven, it should also be mouse-accessible. If the user hovers over options in the slash command menu, use a **`pointer`** cursor on each option. This indicates that suggestions (e.g. turning text into an image block or a heading) are selectable by click as well. Each menu item should also highlight on hover (e.g. a subtle background highlight) to aid recognition. The cursor returns to the **`text`** I-beam once an option is selected and the focus goes back to typing.
- **“+” New Block Button:** An **Add Block** button (“+”) appears to the left of an empty line as a mouse-friendly way to insert new blocks. This button should always use a **`pointer`** on hover to denote a clickable action. Clicking the "+" opens the same block selection menu as the slash command. Ensure the cursor remains **`pointer`** when hovering over menu items from this button as well. Also provide a tooltip on the "+" (e.g., “Add a new block”) on hover for clarity – this benefits discoverability for new users. After clicking an option, the focus goes into the newly inserted block (with **`text`** cursor if it's a text-based block).
- **Inline Interactions and Links:** Within block content, if there are inline interactive elements (e.g. wiki links like [[Some Topic]], tag chips, or embedded media), use a **`pointer`** when hovering those items to indicate they can be clicked or opened. For example, a wikilink might open a note preview on click – the **`pointer`** cursor and perhaps an underline on hover will cue the user. Consistently apply **`pointer`** to any icon or button that appears inside a block (such as a “remove” X on an embedded tag or a checkbox in a to-do item) so the user knows those are actionable. Conversely, non-interactive inline elements (plain text, static icons) should retain the default arrow or **`text`** cursor as appropriate, so as not to mislead.

## **Community Engagement Elements (Badges, Ranks, Hover States)**

- **Achievement Badges:** Community contribution badges (earned for milestones like uploads or quality contributions) should clearly signal interactivity if clickable. These badges – e.g. *Accomplishment Badges* like "Scribe" (for 10 uploads) or "Archivist" (100 uploads), and *Discourse Badges* like "Sage" (for highly-rated annotations) – are displayed on user profiles or posts as icons. When a badge icon is hoverable (for example, if clicking it shows a description or leads to a page listing all badges), use a **`pointer`** cursor on hover. Also implement a hover tooltip or popover that briefly explains the badge (e.g., “**Sage** – awarded for highly-rated annotations”), so users understand its meaning without clicking. The combination of the pointer cursor and a tooltip on hover provides both an interactive cue and immediate context, which is good for accessibility. If the badge is merely decorative (not clickable), then no pointer should appear; however, a tooltip is still recommended on hover for clarity.
- **User Ranks & Titles:** Users earn ranks/titles as their reputation grows (e.g., progressing from *Neophyte* to *Adept* to *Magus* in the community hierarchy). If these rank labels or icons next to usernames are interactive (say, clicking a rank could filter users by rank or show an explanation of the rank system), they should use a **`pointer`** on hover. As with badges, provide a tooltip on hover (e.g., “Rank: Adept – mid-level contributor”) to explain the title in accessible text. If the rank display is not clickable (just informational text), use the normal default cursor so as not to imply an action. In either case, ensure the rank is visually distinct (perhaps styled in the accent color or with an icon) and that on hover it might subtly highlight (even if not clickable) to draw attention to the user’s status. This visual feedback on hover (a slight color change or underline) helps users notice these elements and ties into the platform’s design theme.
- **Interactive Icons (Votes & Actions):** Common community actions like upvote/downvote arrows, “coin” reputation icons, follow buttons, or share icons should all use the **`pointer`** cursor on hover. This consistent rule tells the user that these small icons are buttons. For instance, if there’s an upvote arrow on an annotation, hovering it turns the cursor to **`pointer`** and perhaps changes the arrow color (using the old-gold accent for hover/active states) to indicate it can be clicked. Provide clear hover states: icons can slightly enlarge or glow with the accent color on hover to confirm they are actionable. Remember to also show active state differences (e.g., a filled-in icon when something is liked) but maintain the **`pointer`** on hover even in active state so the user knows they can toggle it off if applicable.
- **Profile Links and Avatars:** Usernames and profile pictures should be treated as interactive links to user profiles. As such, they get a **`pointer`** cursor on hover (and typically an underline or highlight for text). Hovering over a username might also show a mini-profile card – if so, the cursor remains **`pointer`** while over the name or avatar. The mini-profile itself, if it contains interactive elements (e.g., a “Follow” button), should follow the same cursor rules (pointer for that button, etc.). This ensures all community engagement elements adhere to a consistent pattern: anything clickable gets a pointer cursor and a visual highlight on hover, improving discoverability and conforming to user expectations.

## **Navigation & Filtering Systems (Lens Selectors, Drop-Downs, Sidebars)**

- **Lens Selector Controls:** The platform’s **Lens** system allows users to view content through different thematic filters (e.g., an *Astrological Lens* or *Elemental Lens* for the graph). Lens selectors should be very obvious as interactive controls. If implemented as a drop-down menu of lens options, the drop-down trigger (e.g., a button labeled “Lens: Elemental”) should use a **`pointer`** on hover. If implemented as a set of toggle buttons or icons for each lens, each option should show **`pointer`** on hover and a distinct visual state when active. For example, an active lens button might be highlighted in the accent color or with a pressed-in appearance to indicate it’s selected. On hover over any lens option (selected or not), consider a slight accent outline or tooltip (“Switch to Qabalistic Lens”) to reinforce interactivity. Using the old-gold accent for the hover or active state of these lens buttons will tie into the overall UI theme. Once clicked, the lens changes the view; the cursor remains **`pointer`** during the click on the control and then may revert to default when back over the static canvas or content.
- **Filter Drop-Downs & Menus:** Any drop-down used for navigation or filtering (such as sorting options, account menu, or advanced search filters) should consistently use **`pointer`** when hovering over the drop-down trigger and its list items. For example, a **Filter** drop-down in the library might say “Sort by: Date ▼” – hovering this should show **`pointer`**, and clicking it shows a list of options (Date, Title, Author, etc.). Each option in the list should highlight on hover and use the **`pointer`** cursor, indicating you can click to select it. After selection, the drop-down closes and the cursor returns to whatever is appropriate at the new pointer location. Ensure keyboard accessibility by allowing focus on the drop-down and using standard focus indicators, but visually for cursor users, the pointer and highlight are key. Also, provide a clear indication of the current selection (e.g., a checkmark or bold text in the menu) so users understand which filter is active without relying solely on cursor hover.
- **Sidebar Navigation Links:** The sidebar (e.g., for library sections, personal notebooks, or guild forums) will contain a list of navigational items. All clickable items in the sidebar should show a **`pointer`** on hover. These include links to collection pages, category labels, or expandable sections. For instance, if the sidebar has an item “Alchemy Texts”, hovering that list item changes the cursor to **`pointer`** and perhaps underlines or highlights the text to signal it’s a link. If sidebar sections are collapsible (with a small arrow or +/– icon), that collapse/expand icon is also interactive: use **`pointer`** on the icon (and the icon could rotate on click). Non-interactive headers or labels in the sidebar (for example, a plain title like “Collections” that isn’t clickable) should retain the default cursor to differentiate from clickable entries. Maintain sufficient spacing around sidebar links so that users can easily target them with the cursor (improving Fitts’ Law compliance and overall usability). Additionally, if the sidebar contains a search input field, that field should show a **`text`** cursor when interacted with, as it’s for text entry.
- **Toggle Buttons and Checkboxes (Filters):** In various filtering interfaces (for example, toggling show/hide of certain node types on the graph, or checkboxes to filter search results), apply the **`pointer`** cursor to the clickable labels or custom toggle controls. If the UI uses custom-styled switches or pill buttons for filters, ensure they behave like buttons: **`pointer`** on hover and a visible state change when toggled on/off. This consistency helps users quickly grasp that these are interactive toggles. Standard HTML checkboxes or radio buttons will typically default to **`pointer`** on the clickable area automatically, but if they are hidden and replaced with a styled element, explicitly set the cursor. Always accompany visual changes (like a filled checkbox or highlighted toggle) with the cursor change so the user gets immediate feedback. For example, a “Show orphan nodes” toggle in the graph filter panel should have a pointer on hover and a clear on/off state change when clicked (with maybe a color accent when on).
- **Global Navigation & Buttons:** Any top navigation bar elements (e.g., a home icon, profile menu, notifications bell) should follow the same rule: **`pointer`** on hover. Even if they are icons without text, the pointer cursor plus a tooltip (e.g., “Home”, “Profile”) on hover makes their purpose clear. If there is a hamburger menu or side panel toggle for mobile, that icon should use **`pointer`** as well. Essentially, throughout all navigation, the rule is: if it’s clickable, use **`pointer`**; if it’s an input for text, use **`text`**; if it’s draggable, use **`grab`**/**`grabbing`**. Any purely static text or icons (no action) should use the default arrow so as not to confuse the user.

## **Accessibility and Consistency Considerations**

- **Consistency Across the UI:** Use the same cursor conventions everywhere to meet user expectations. For example, a link in a text block, a button in a dialog, and a clickable icon in the sidebar all should use the **`pointer`** cursor on hover. This uniformity prevents confusion – once users learn the cursor signals (arrow vs. hand vs. I-beam), they can predict what will happen. It also aligns with general web standards (links and buttons use the hand cursor). Similarly, text fields or areas always use the **`text`** cursor, reinforcing that they are for typing. By keeping these rules consistent, the interface becomes more learnable and feels professional.
- **Visual Feedback Beyond Cursor:** While cursor changes are a great immediate indicator, they should be complemented with other visual feedback for clarity and for users who may not see the cursor. Leverage the platform’s design language (colors, highlights) for hover and active states on interactive elements. For example, menu items might get a slight background shade on hover, buttons might darken or glow, and active selections might be bold or accented. This redundancy (cursor change + visual highlight) makes the UI more intuitive. It also aids users with motor or visual impairments – for instance, a user with low vision might notice a button’s color change more than the small cursor shape change.
- **Touch and Keyboard Accessibility:** Remember that not all users have a mouse cursor. Touchscreen users and keyboard-only users won’t see a hover cursor at all. Therefore, ensure that critical interactions do not depend solely on hover. All interactive elements should be reachable and usable via keyboard (tab focus and Enter/Space activation), and focus states should be clearly visible (e.g., a focus ring or outline). The design should meet accessibility guidelines so the platform remains usable by individuals with disabilities. For touch users, provide alternative cues like active state changes on tap and consider touch-friendly hit areas. In practice, this means the cursor rules here are part of a broader strategy: for cursor users we give a clear pointer/grab/text cue, and for non-cursor interactions we rely on equivalent visual cues and ARIA roles/hints. Consistently following these cursor rules, alongside tooltips and focus indicators, will ensure the Digital Grimoire’s UI feels intuitive, responsive, and inclusive to all users.
